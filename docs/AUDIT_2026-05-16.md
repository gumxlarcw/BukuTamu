## 1. Executive Summary

The Bukutamu system is the BPS Maluku Utara visitor-book and PST queueing platform, consolidated on 2026-05-16 into a monorepo containing three deployables: a PHP **backend** (CodeIgniter 3 + HMVC), a React/TypeScript **frontend** (Vite 8 + React 19), and a Node **print server** that runs locally on each kiosk PC.

**Overall health:** **Good.** The codebase shows deliberate engineering across three notable axes: (a) role-and-service authorization is mirrored carefully between PHP (`Api_base.php`) and TypeScript (`role-access.ts`), (b) the finalization gate is layered defense-in-depth across three controllers, (c) the print server is intentionally decentralized to each kiosk PC, which removes a long-distance network failure mode.

**Key risks identified:**

- 🔴 **`/api/audit` has no role gate** — any authenticated user (including receptionist) can read the full admin audit log, leaking admin actions and IP addresses.
- 🔴 **Legacy CI3 web flow (`Selamat_datang::send_to_printer`) targets `localhost:5000`** — wrong port (5300 is correct). Dead-code footgun; if anyone reactivates the old path, prints will be sent to `pds-backend` instead of the print server.
- 🟡 **`/api/evaluations/{id}` POST is intentionally unauthenticated** and deletes existing rows before insert. Any actor knowing a visit ID in `menunggu_evaluasi` state can clobber a real evaluation. Documented in code but worth elevating.
- 🟡 **No client-side role gate** on admin routes — backend returns 403 correctly, but a logged-in receptionist can navigate to `/admin/users` and see an empty UI before requests fail.
- 🟡 **Two parallel auth systems coexist**: the new JWT API and the legacy CI3 session admin panel under `/admin/*` (PHP routes). The legacy panel still routes through Apache.
- 🟢 **Phase 9 consolidation cleanup is pending** — legacy controllers (`Evaluasi.php`, `selamat_datang`, `recognize`, etc.) are dead code but remain reachable.

**Recommended action queue** (top 3): gate `/api/audit` to admin+; delete the legacy CI3 web controllers; add `requireRole()` guard on client-side admin routes.

### 2.3 Tech stack

**Backend:**
- PHP / CodeIgniter 3 + HMVC (`MX_Controller`)
- Database: MySQL `db_tamdes` (name preserved during rename for prod safety)
- Auth: custom JWT (HS256) in httponly cookie + database-backed CI sessions for legacy `/admin/*`
- Autoload: `database`, `session`, `url` helper only — narrow surface
- Libraries: `JWT_Helper.php`, `Layout.php`
- Per-module models live inside HMVC modules; no top-level `application/models/`

**Frontend:**
- React 19, TypeScript ~5.9, Vite 8, Tailwind v4
- State: `@tanstack/react-query` v5
- Router: `react-router-dom` v7
- HTTP: `axios` v1.13 with `withCredentials: true`
- Toasts: `sonner`
- Calendar: `@fullcalendar/*`
- Face recognition: `face-api.js` v0.22.2 loaded via CDN at `index.html`
- `.npmrc legacy-peer-deps=true` — Tailwind 4 + Vite 8 ecosystem still settling

**Print:**
- Node + `express` 4.19 + `escpos` 2.5 + `escpos-usb` 3.0
- ESC/POS thermal protocol, USB VID/PID configurable via env (default STMicro 0x0483:0x5840)

### 2.5 Service taxonomy (mirrored FE + BE)

| Group        | Services                                                              | Role required          | Finalization status      | Sarana codes        |
|--------------|-----------------------------------------------------------------------|------------------------|--------------------------|---------------------|
| **SKD (4)**  | Perpustakaan, Konsultasi Statistik, Rekomendasi Kegiatan Statistik, Penjualan Produk Statistik | `petugas_pst` | `menunggu_evaluasi` (then tablet evaluasi → `selesai`) | 1, 2, 4, 9, 16, 32 |

**Strategy C (cross-layanan rejection):** A visit may NOT mix groups. Enforced in `Api_base::validate_no_cross_layanan` and `lib/role-access.ts::isCrossLayanan`. UI disables the Lock-icon button on incompatible service combinations.

**Bypass roles:** `superadmin`, `admin`, `operator` (legacy) — bypass layanan-role checks but still pass through standard auth and finalization gates.

## 3. Backend Endpoint Inventory

### 3.1 Routing model

All routes are declared explicitly in `application/config/routes.php`. No auto-discovery. Two surfaces coexist:

- **JSON API** under `/api/*` — used by the React frontend, JWT-cookie auth.
- **Legacy web views** under `/`, `/layanan`, `/selamat_datang`, `/admin/*`, `/antrian/cetak/*`, `/evaluasi/*` — CI3 session auth, server-rendered HTML, **superseded** by the React kiosk + admin pages but routes still live.

### 3.2 JSON API endpoints (the active surface)

Legend:
- **Auth:** `none` = unauthenticated, `JWT` = JWT cookie required, `JWT+role` = JWT plus minimum role.
- **Side effects:** key mutations beyond the obvious CRUD.

```
   │                │   if gaps: ProfileGapsModal → /api/kiosk/profile-update
```

**Server-side actions inside `POST /api/kiosk/register`:**
1. `validate_no_cross_layanan` → 400 if SKD + Resepsionis mixed
2. `validate_sarana_for_layanan` → 400 if sarana code not whitelisted for group
3. Dedup: same `nama` + `notel` + today → return existing visit
4. `LOCK TABLES` (tamdes_buku, tamdes_kunjungan, tamdes_responden_tahunan)
5. `id_user = MAX+1` (starts 8200001)
6. base64-decode `foto`, JSON-encode `face_descriptor`
7. Insert guest + visit
8. `generate_queue_number` → DTSEN gets prefix `D`, others first letter of layanan
9. `UNLOCK TABLES`
10. Return `{id_kunjungan, id_user, nomor_antrian}`

```
[Kiosk arrival creates visit with status='antri', layanan=SKD]
                          │
                          ▼
              /admin/consultations (PST officer)
                          │

                          │   → insert N new rows (one per data need)
                          │   → next_status_after_completion() = 'menunggu_evaluasi'
```

**Three defense-in-depth status finalization gates:**

1. **Visit/Consultation `status` PUT**: soft-correct SKD `selesai` → `menunggu_evaluasi` unless caller is bypass role (admin/superadmin/operator).
2. **`require_layanan_role`** on any transition to `selesai`/`menunggu_evaluasi`: enforces PST officer can finalize PST visits, receptionist can finalize receptionist visits.
3. **`POST /api/evaluations/{id}` gate**: rejects unless visit status ∈ {menunggu_evaluasi, selesai}. The *only* defense for the unauthenticated tablet endpoint.

## 6. Cross-Cutting Concerns

### 6.1 Authentication & authorization

**Two parallel systems:**

| System          | Where it lives                          | Auth mechanism                       | Scope                |
|-----------------|-----------------------------------------|--------------------------------------|----------------------|
| Modern (active) | `Api_base` + descendants                | JWT (HS256) in httponly `jwt_token` cookie, 4hr TTL | `/api/*` routes |
| Legacy          | `Admin` controller                      | CI3 database-backed `ci_session`     | `/admin/*` HTML routes |

**Role hierarchy** (from `Api_base::require_role`):

| Role             | Level | Notes |
|------------------|-------|-------|
| `operator`       | 1     | Legacy, bypass on layanan checks |
| `resepsionis`    | 1     | Layanan: Lainnya, Keperluan Pimpinan |
| `petugas_pst`    | 1     | Layanan: 4 SKD + DTSEN |
| `admin`          | 2     | Bypass + delete authority |
| `superadmin`     | 3     | User management |

**Layanan-role guard** (`require_layanan_role`): when transitioning a visit to `selesai`/`menunggu_evaluasi`, the caller's role must match the visit's service group. Bypass roles (admin/superadmin/operator) override. Mirrored in `lib/role-access.ts::canFinalizeLayanan`.

### 6.2 Rate limiting

- **Login**: 5 failed attempts per IP per 15 min → 429 + lockout message. Tracked in `tamdes_login_attempts`.
- **No other endpoint is rate-limited.** Notably `/api/kiosk/*` (registration, profile-update) and `/api/evaluations/{id}` POST are unauthenticated *and* unlimited.

### 6.3 CSRF posture

- All mutating API calls are POST/PUT/DELETE with JSON body (`application/json`).
- No CSRF token; relies on **SameSite=Strict** cookie attribute.
- CORS allowlist: `http://localhost:5173` (dev) + `FRONTEND_URL` env (prod). Restrictive.
- `Access-Control-Allow-Credentials: true` is set.

Posture: cookie + SameSite-Strict + restrictive CORS is acceptable for this threat model. A determined attacker who could phish a user onto a CORS-allowed origin would still need to bypass SameSite.

### 6.4 Audit logging (`tamdes_audit_log`)

Written by:
- `Auth::login` / `Auth::logout` (action: `login` / `logout`)
- `Visits::detail` DELETE (`delete`, captures pre-state)
- `Visits::status` (`update_status`, `from→to`)
- `Visits::service` (`update_service`, diff)
- `Guests::detail` PUT (`update`, diff)
- `Guests::detail` DELETE (`delete`)
- `Users::*` (`create`, `update`, `delete`, `change_password`)
- `Consultations::call` (`call_queue` / `call_queue_failed`)
- `Consultations::data` POST (`save_consultation`)
- `Dtsen::data` POST (`save_dtsen`)

**Read access:** `/api/audit` — JWT only, no role gate. **🔴 finding.**

### 6.5 Status finalization gates (memory says 3 defenses)

| Gate | Where | What it does |
|------|-------|--------------|
| 1 | `Visits::status` PUT + `Consultations::detail` PUT | Soft-correct `selesai` → `menunggu_evaluasi` for SKD when caller is not bypass role |
| 2 | `require_layanan_role` invoked on transitions to `selesai`/`menunggu_evaluasi` in `Visits::status`, `Visits::service`, `Visits::summary`, `Consultations::detail`, `Consultations::data`, `Dtsen::detail`, `Dtsen::data` | Ensures caller role matches service group |
| 3 | `Evaluations::detail` POST | Visit must be in `menunggu_evaluasi` or `selesai`. Sole defense for unauthenticated tablet endpoint. |

### 6.6 Face recognition pipeline

Constants in `lib/face-detection.ts`:
- `inputSize: 320, scoreThreshold: 0.5` (face detector)
- Match threshold: **0.55** (Euclidean distance, lower = stricter)
- Match margin: **0.08** (best vs 2nd-best gap — rejects ambiguous matches)
- Sampling: **5 descriptors averaged** to reduce per-frame noise
- Warmup: 600ms (from memory)

Models loaded from `/models/` (vite public dir), face-api.js loaded via CDN at `index.html`.

### 6.7 Print payload contract

```jsonc
{
  "no": "K001",              // alias for compat with legacy
  "nomor_antrian": "K001",   // canonical
  "nama": "Budi",
  "jenis_layanan": "Konsultasi Statistik",
  "date_visit": "2026-05-16 10:42:13"
}
```

`print/server.js` accepts either `no` or `nomor_antrian` — returns 400 if both missing. Truncates name + layanan to 32 chars (thermal printer width).

### 6.8 Error handling & silent failures

- **`apiClient.interceptors.response`** silently swallows non-401 errors at the boundary; pages must `try/catch` their own mutations.
- **`AuthProvider.doLogout`** swallows `authApi.logout()` rejection silently (`try { ... } catch {}`).
- **`Consultations::call`** intentionally fails LOUDLY (502 + audit) when dashboard-pst is unreachable — anti-silent-failure (good).
- **`Selamat_datang::send_to_printer`** (legacy) catches and logs without surfacing to user — but dead code.

### 6.9 SQL injection posture

- All API controllers use CI3 query builder (`$this->db->where/get/insert/update`) which auto-escapes.
- **Exceptions:**
  - `Responden::_base_query` builds a `FROM (subquery)` string with `intval($year)` and `escape_like_str($q)` — safe but inconsistent.
  - `Responden::_skd_clause` uses `addslashes($svc)` where `$svc` comes from a hardcoded whitelist (`$eligible_services`) — currently safe, brittle if list ever becomes dynamic.
- No raw `$this->db->query(...)` with user input found.

### 6.10 Secrets management

- `.env` is git-ignored; `.env.example` is committed.
- `JWT_SECRET` must be ≥32 chars; `JWT_Helper` falls back to a deterministic per-install hash with `log_message('error')` — never crashes but logs an error.
- `ADMIN_PASSWORD_HASH` is bcrypt (CI3 admin) or accessible to both legacy `Admin` controller and modern `Auth::login` fallback.
- DB credentials via env (`DB_HOSTNAME`, `DB_USERNAME`, `DB_PASSWORD`, `DB_DATABASE`).

## 7. Findings & Recommendations

### 🔴 Critical

#### C1 — `/api/audit` has no role gate
**Where:** `application/modules/api/controllers/Audit.php:9` (only `require_auth()`)
**Impact:** Any authenticated user — including `resepsionis`, `petugas_pst`, `operator` — can read the full audit log, including admin/superadmin actions, IP addresses, and payload diffs (which may include guest PII).
**Recommendation:** Add `$this->require_role('admin');` after `$this->require_auth();` in `Audit::index`. Optionally also gate the FE route `/admin/audit` to admin+ in `AdminLayout`.

#### C2 — Legacy print path points to wrong port
**Where:** `application/modules/selamat_datang/controllers/Selamat_datang.php:116` — `$url = 'http://localhost:5000/print'`
**Impact:** Currently dead code (React kiosk doesn't use this route). If anyone reactivates the legacy `/selamat_datang/masuk` flow, prints will be sent to `pds-backend` (port 5000) instead of the print server (port 5300). Worst case: pds-backend logs garbled requests.
**Recommendation:** Delete the entire legacy `selamat_datang` module (and `recognize`, `layanan`, top-level `Evaluasi.php`) as part of Phase 9 consolidation cleanup. They are not called by the React FE.

#### C3 — Unauthenticated profile-update endpoint
**Where:** `Kiosk::profile_update` (`POST /api/kiosk/profile-update/{id_user}`)
**Impact:** Anyone on the public internet who guesses an `id_user` (sequential starting at 8200001) can patch any guest's `email`, `notel`, `pendidikan`, `pekerjaan`, etc. No rate limit. This is the only endpoint that mutates `tamdes_buku` without authentication.
**Recommendation:** Either (a) require a "session token" tied to a recent face-match (mint a short-lived signed token from `/api/kiosk/face-data` workflow and require it here), or (b) accept the risk explicitly and add rate limiting (e.g., 1 update per id_user per 5 min from same IP).

### 🟡 Medium

#### M1 — `/api/evaluations/{id}` POST is fully unauthenticated and destructive
**Where:** `Evaluations::detail` POST branch
**Impact:** Any actor with a visit ID in `menunggu_evaluasi` state can submit (and clobber) evaluations. The endpoint **deletes existing rows before insert**, so a legitimate evaluation can be overwritten. Documented as intentional ("tablet kiosk, gate ini adalah satu-satunya defense") but worth elevating to defense-in-depth.
**Recommendation:** Mint a short-lived signed token in `/api/evaluations/pending` response and require it on POST. Token bound to `id_kunjungan` + `iat`, 10-min expiry. Tablet stores token in memory; no backend session needed.

#### M2 — Two parallel auth systems coexist; legacy `/admin/*` is still routable
**Where:** `application/modules/admin/controllers/Admin.php`
**Impact:** Old CI3 admin panel still works. Non-production fallback allows `admin/admin123` if env not set. URL pattern conflicts with React `/admin/*` (Apache vhost likely handles this, but worth verifying).
**Recommendation:** Either (a) delete the legacy `admin` module entirely (Phase 9), or (b) gate its `index()` to return 410 Gone + redirect to React `/login`. Verify Apache vhost rewrites don't expose it.

#### M3 — Client-side has auth gate but no role gate on admin routes
**Where:** `layouts/AdminLayout.tsx` (only `if (!user) return <Navigate>`)
**Impact:** A `resepsionis` who navigates to `/admin/users` sees an empty page (the BE returns 403, the FE doesn't catch and redirect). Confusing UX but not a security issue (BE is authoritative).
**Recommendation:** Add `<RequireRole min="superadmin">{children}</RequireRole>` wrapper for `/admin/users`, `<RequireRole min="admin">` for `/admin/audit` after fixing C1.

#### M4 — Unauthenticated `/api/kiosk/face-data` and `/api/kiosk/guest-list` leak PII
**Where:** `Kiosk::face_data`, `Kiosk::guest_list`
**Impact:** Anyone hitting the public domain can download the full list of guest names + (for face-data) their biometric descriptors and id_users. The face descriptor cannot be reversed to a photo but allows tracking the same person across systems.
**Recommendation:** Accept the risk (kiosk needs them, no auth available), but consider (a) network-level restriction so only kiosk PCs can reach these endpoints (Apache ACL by source IP), and (b) excluding inactive/old guests from the dump.

#### M5 — `/api/kiosk/visit` (returning guest) has no dedup
**Where:** `Kiosk::visit`
**Impact:** A returning guest at the kiosk can press the button N times, creating N visit records in quick succession. `Kiosk::register` has a 30-second dedup but `Kiosk::visit` does not.
**Recommendation:** Apply the same dedup as `register`: same id_user + today → return existing visit.

#### M6 — `Guests::detail` DELETE doesn't cascade
**Where:** `Guests::detail` DELETE branch
**Impact:** Deleting a guest leaves orphaned rows in `tamdes_kunjungan`, `konsultasi_pengunjung`, etc. (FK is `id_user` but no FK constraint observed at DB level either). Reports like `/api/responden` will show stale data, and joins to `tamdes_buku` will return NULL for these rows.
**Recommendation:** Either cascade-delete the visits (mirror `Visits::detail` DELETE chain), or — more conservatively — refuse to delete a guest with any visits and return 409.

#### M7 — Inconsistent API client modularization
**Where:** `src/api/` has 10 modules but admin pages directly use `apiClient` for `/api/users`, `/api/audit`, `/api/queue-stats`, `/api/responden`, `/api/guests/{id}/visits`.
**Impact:** Endpoint contracts drift across pages; harder to refactor.
**Recommendation:** Add `users.ts`, `audit.ts`, `queueStats.ts`, `responden.ts` modules with typed return shapes. No behavior change.

### 🟢 Nits / observations

#### N1 — Race in `Guests::index` POST id_user generation
`Guests::index` POST does `MAX(id_user)+1` without `LOCK TABLES` (unlike `Kiosk::register`). Two concurrent admin creates could collide. Low traffic makes this academic, but inconsistency is real.

#### N2 — `evaluationsApi.getSummary` references type before declaration
`evaluations.ts:47` uses `EvaluationSummary` as generic; the type is declared at line 68. TS hoists types so this compiles, but ordering is unconventional.

#### N3 — `Selamat_datang::recognize` uses guest name as primary key
Legacy code matches by `nama` string equality, which is brittle (duplicates, whitespace). The new API uses `id_user` correctly. Another reason to delete legacy modules.

#### N4 — `Responden::_skd_clause` uses `addslashes` instead of CI3 escaping
Currently safe because `$eligible_services` is a hardcoded array, but `addslashes` is the wrong primitive for SQL. If services ever become user-configurable, this is an injection vector.

#### N5 — `Dashboard::stats` rebuilds queries instead of caching subquery

If print fails (USB unplugged), the toast appears but no automatic retry. Acceptable for a kiosk where the user can press "Reprint" button — `TicketPage` provides one.

#### N7 — Hardcoded service list in `Services::index`
The service registry is in PHP code (not a DB table). Adding a service requires a PHP deploy + matching TS deploy in `role-access.ts`. The "FE-BE parity" memory note covers this.

#### N8 — `print/server.js` allows `Access-Control-Allow-Origin: *`
Reasonable for localhost-only access, but if the print server ever binds to a non-loopback interface, this becomes a real CORS hole. Consider binding explicitly to `127.0.0.1` in `app.listen`.

### 9.5 NEW finding discovered during smoke testing

#### M8 — Silent FK constraint failure in `Kiosk::register` and `Kiosk::visit`

**Where:** `application/modules/api/controllers/Kiosk.php` — both `register()` and `visit()` end with `$this->db->insert('tamdes_kunjungan', $visit_data); $id_kunjungan = $this->db->insert_id();` and **never check whether the insert succeeded**.

**Impact:** `tamdes_kunjungan` has FK constraint `id_user → tamdes_buku(id_user)` (this audit's section 2.4 schema list was incomplete — corrected here). When the FK constraint rejects an insert, `insert_id()` returns `0`, but the endpoint still returns `{"success": true, "data": {"id_kunjungan": 0, ...}}`. The frontend then redirects to `/kiosk/ticket/0` and 404s, while the user sees "Pendaftaran berhasil" toast. The CI3 error log captures the FK error, but the user/operator gets no signal.

**Reproducer:** Discovered by accident — `POST /api/kiosk/visit {id_user: 99999999, ...}` returns 201 success with id_kunjungan=0, no row created.

**Recommendation:**
```php
$this->db->insert('tamdes_kunjungan', $visit_data);
$id_kunjungan = $this->db->insert_id();
if (!$id_kunjungan) {
    $err = $this->db->error();
    log_message('error', 'Kiosk::visit insert failed: ' . print_r($err, true));
    $this->json_response([
        'success' => false,
        'message' => 'Gagal membuat kunjungan. ID tamu tidak valid atau database error.',
    ], 500);
}
```
Apply to both `register()` (after both `tamdes_buku` and `tamdes_kunjungan` inserts) and `visit()`.

**Severity:** 🟡 Medium — production FE never sends bogus id_user, so no live impact today; but the silent failure violates the codebase's own anti-silent-failure principle and would mask data-corruption bugs if any.
